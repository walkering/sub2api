## 上下文

当前仓库已经具备两块可复用能力：
- 账号编辑 / 批量编辑中已经支持模型白名单和模型映射，并最终写入账号 `credentials.model_mapping`。
- 后端已有账号批量更新基础能力，也已经存在“按分组缩小账号执行范围”的相关语义。

缺失的是：在“分组管理”页面，管理员无法直接以分组为入口，对该分组内账号统一设置模型白名单或模型映射。现在只能切去账号页筛选、勾选、再批量编辑，操作链路长且容易漏选。

## 目标 / 非目标

**目标：**
- 在分组管理中新增针对“分组内账号”的模型白名单 / 模型映射批量设置入口。
- 复用现有账号编辑 / 账号批量编辑的模型限制表单和校验规则。
- 让后端按分组解析目标账号集合，避免前端为了批量更新而预先拉取全量账号 ID。
- 保持账号运行时模型限制语义不变，继续依赖账号现有 `credentials.model_mapping` 生效。

**非目标：**
- 不新增用户级模型限制，也不修改用户管理接口。
- 不引入新的运行时映射层级，不调整渠道级、账号级、分组默认映射的执行顺序。
- 不重做现有账号批量编辑功能；本次只补充分组管理入口。
- 不扩展到分组内其他批量字段，本次聚焦模型白名单与模型映射。

## 决策

### 决策 1：分组管理走“按分组批量更新账号”的独立入口

新增一个面向分组管理的批量更新入口，输入至少包含 `group_id` 和模型限制配置；服务端自行查询该分组下账号并执行批量更新。

原因：
- 分组页不应该为了一个批量动作先把该分组所有账号 ID 拉到前端。
- 现有 `/admin/accounts/bulk-update` 更适合“我已经选中了一批账号”的场景，而分组页是“我选中了一个分组”。
- 独立入口可以返回更贴近业务的结果，如命中账号数、成功数、失败数。

备选方案：
- 前端先查询分组账号列表再调用已有 bulk-update：实现快，但前端负担更重，且对大分组不友好。
- 直接复用分组更新接口：会把“修改分组自身字段”和“修改分组内账号字段”混在一起。

### 决策 2：底层仍然复用账号批量更新的模型限制写法

无论入口来自账号页还是分组页，最终都统一写入账号 `credentials.model_mapping`，白名单模式继续转换为“同名映射”，映射模式继续使用现有映射对象。

原因：
- 现有账号运行时逻辑已经依赖 `credentials.model_mapping`，不需要新增新的持久化结构。
- 这能最大化复用现有白名单选择器、映射校验和运行时行为。

备选方案：
- 为分组页单独新增专用字段：会造成同一能力两套存储语义。

### 决策 3：分组批量表单与账号批量编辑保持一致

分组管理中的批量设置弹窗复用现有账号批量编辑中“模型限制”部分的交互，包括：
- 白名单 / 映射双模式切换
- 模型白名单选择器
- 映射规则增删与冲突校验
- OpenAI 自动透传等现有禁用提示

原因：
- 用户期望的是“同样的能力，只是入口换到分组管理”。
- 统一交互可以减少认知成本，也避免同一模型限制在两个入口下行为不一致。

备选方案：
- 在分组页重写一个轻量表单：短期可做，但校验和文案很容易与账号页分叉。

### 决策 4：目标范围定义为“当前分组下的账号绑定关系”

批量设置作用对象为当前分组已绑定的账号；如果账号同时属于多个分组，本次更新仍然修改该账号本身，因为模型限制是账号级配置。

原因：
- 账号模型限制本来就是账号自身属性，而不是“某个分组内的局部视图”。
- 这与现有账号批量编辑语义一致：一旦保存，就是修改账号本体。

备选方案：
- 仅在当前分组上下文临时生效：需要新增新的“分组-账号局部覆盖层”，超出本次范围。

## 风险 / 权衡

- [账号同时属于多个分组，管理员可能误以为修改只影响当前分组] → 在弹窗和确认文案中明确说明“修改的是账号本身，对该账号在其他分组中的使用也会生效”。
- [分组页和账号页各维护一套模型限制表单会导致行为分叉] → 共享现有账号批量编辑中的模型限制组件和校验逻辑。
- [大分组批量更新可能部分失败] → 接口返回命中数、成功数、失败数，并复用现有批量更新结果结构或兼容摘要。

## Migration Plan

1. 新增分组管理侧的账号模型限制批量更新接口，并复用现有账号模型限制写入逻辑。
2. 在分组管理页面增加“批量设置账号模型限制”入口与弹窗。
3. 补充前后端测试，验证分组范围解析、模型限制写入和结果回显。
4. 回滚时只需移除分组页入口和后端新接口；账号已有模型限制数据结构保持不变。
