## 上下文

当前启动入口仅提供 `-setup` 与 `-version` 两个命令行参数，配置加载则固定依赖 `DATA_DIR`、`/app/data`、当前目录、`./config`、`/etc/sub2api` 等搜索路径。这个模式对 Docker 场景足够，但对二进制部署、多实例共机部署、从任意工作目录调用发布包，以及临时验证某份配置文件的场景不够直接。

本次变更同时影响命令行入口和配置模块：启动前要先解析 `-c` 参数，而后续的 `LoadForBootstrap()`、`Load()` 与 `GetServerAddress()` 都必须使用同一份配置定位结果，否则会出现“setup 阶段读一份配置、正式启动读另一份配置”的不一致。

## 目标 / 非目标

**目标：**
- 支持通过 `./sub2api -c config.yaml` 显式指定配置文件。
- 让 bootstrap 配置读取、正式配置读取、setup 阶段监听地址读取共享同一配置文件选择结果。
- 未传入 `-c` 时保持现有搜索路径和环境变量行为不变。
- 当指定路径不存在或不可读时，返回明确错误，避免静默回退到其他配置文件。

**非目标：**
- 不改造配置文件格式。
- 不移除现有 `DATA_DIR`、默认搜索路径或环境变量覆盖机制。
- 不支持一次传入多个配置文件或目录。
- 不引入热加载、watch 或运行时切换配置文件能力。

## 决策

### 决策 1：在启动入口增加短参数 `-c`，并提供长参数别名 `-config`
- 原因：用户明确需要 `./sub2api -c config.yaml` 这种用法；同时补充 `-config` 便于自解释。
- 方案：在 `backend/cmd/server/main.go` 中将两个 flag 绑定到同一个字符串变量。
- 备选方案：
  - 仅支持 `-config`：可读性更好，但不满足用户期望的短参数形式。
  - 仅支持环境变量：无法满足命令级显式指定配置的需求。

### 决策 2：在 `config` 包中引入“显式配置文件路径”设置点，由所有加载函数统一读取
- 原因：`Load()`、`LoadForBootstrap()`、`GetServerAddress()` 目前各自创建/使用 viper 读取配置；如果只在某个调用点单独传参，容易产生行为分裂。
- 方案：在 `config` 包增加一个集中设置的显式路径（例如 setter/getter 或内部共享状态），让 `load(...)` 与 `GetServerAddress()` 都先判断是否存在显式路径；存在时调用 `SetConfigFile(...)`，否则保留原有 `AddConfigPath(...)` 搜索链。
- 备选方案：
  - 给每个函数都加 path 参数：调用链改动更大，且会扩散到 wire/bootstrap/setup 等调用点。
  - 只在主流程中用一次路径，其他地方保留旧逻辑：会导致 setup 地址和正式启动读取不一致。

### 决策 3：显式配置文件路径优先级高于默认搜索路径，但不打破环境变量覆盖
- 原因：`-c` 的目标是指定“读取哪一个配置文件”；环境变量覆盖是现有全局能力，应继续生效。
- 方案：当传入 `-c` 时，viper 直接读取该文件，再继续启用 `AutomaticEnv()` 覆盖。
- 备选方案：
  - 让 `-c` 完全屏蔽环境变量：会破坏现有部署习惯。
  - 传入 `-c` 后仍静默回退默认搜索路径：不利于排错，也削弱显式指定的意义。

### 决策 4：显式路径读取失败时直接报错，不做静默降级
- 原因：`./sub2api -c wrong.yaml` 如果悄悄退回默认配置，用户会误以为已加载目标文件。
- 方案：当 `SetConfigFile()` 指定的文件不存在、不可读或格式错误时，返回明确错误并终止启动。
- 备选方案：
  - 回退默认路径：兼容性更强，但可观测性差，风险高。

## 风险 / 权衡

- [全局共享显式配置路径可能影响测试隔离] → 提供重置能力或在测试中显式清理状态。
- [短参数 `-c` 与未来其他 CLI 参数冲突] → 当前启动入口参数很少，保留 `-config` 作为稳定长名。
- [setup 阶段与正式启动共用配置选择逻辑后，错误会更早暴露] → 这是有意行为，能更早发现错误配置路径。
- [文档中仍可能保留旧启动方式] → 同步更新 README / deploy 文档中的运行示例。

## Migration Plan

- 发布包含新参数的二进制版本。
- 现有部署无需修改即可继续运行，因为未传入 `-c` 时行为保持不变。
- 需要显式指定配置文件的部署可改用 `./sub2api -c /path/to/config.yaml`。
- 若出现问题，可移除 `-c` 参数回退到原有搜索路径启动方式。
