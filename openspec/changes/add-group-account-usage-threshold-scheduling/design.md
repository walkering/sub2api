## 上下文

当前系统已经具备多种账号调度约束能力，包括基础可调度状态、速率限制、窗口费用控制以及粘性会话续用。账号的配额使用量也已经持续写入 `accounts.extra`，包含总额度、日额度和周额度等字段，但分组调度还不能基于这些使用量百分比做“接近上限时停止新调度”的控制。

这次变更横跨分组管理、调度快照、网关粘性会话与管理端展示，且需要和现有“窗口费用达到阈值后仅允许粘性会话继续”的模式保持一致，否则容易出现正在服务中的会话被突然打断，或者账号在不同入口下行为不一致的问题。

## 目标 / 非目标

**目标：**
- 为分组增加账号使用量阈值百分比配置，支持运营按组控制账号何时退出新调度。
- 以账号已有的配额使用数据为基础，稳定判定账号是否达到阈值。
- 达到阈值后仅阻止新调度，不中断仍在进行中的粘性会话。
- 在活跃会话结束后，立即让账号退出当前分组的调度候选，并让后续请求切换到其他可用账号。

**非目标：**
- 不改造现有 usage 计费写入逻辑，不新增独立的使用量统计来源。
- 不强制终止正在进行中的上游请求，也不主动清理现有会话缓存。
- 不引入新的全局账号封禁语义；阈值仅在请求所属分组内生效。
- 不处理“所有账号都达到阈值”时的自动扩容或自动恢复策略，沿用现有无可用账号失败路径。

## 决策

### 决策 1：阈值配置存放在分组层，并按请求分组独立生效

分组新增 `account_usage_threshold_percent` 配置字段，取值范围为 `0 < x <= 100`，空值或 `<= 0` 视为关闭。账号被调度时，系统仅使用当前请求所属分组的阈值进行判断，不把该组配置外溢为全局账号状态。

原因：
- 用户需求明确要求“在分组里设置”控制项。
- 一个账号可以挂到多个分组，分组独立生效能保持配置边界清晰，避免把单组策略误变成全局封禁。

备选方案：
- 配置在账号层：无法满足按分组差异化控制。
- 取账号所属全部分组的最严阈值：实现复杂且会产生跨分组的隐式耦合。

### 决策 2：使用“已配置配额维度中的最大使用比例”判定是否达到阈值

系统从账号现有配额字段中提取可计算的比例：`quota_used / quota_limit`、`quota_daily_used / quota_daily_limit`、`quota_weekly_used / quota_weekly_limit`。仅当分母大于 0 时该维度才参与计算，最终取这些比例中的最大值，与分组阈值进行比较；最大比例大于等于阈值百分比时视为已达阈值。

原因：
- 账号可能同时受总额度、日额度、周额度约束，只看总额度会漏掉短周期封顶风险。
- 取最大比例语义直接，可覆盖“任一已启用配额维度接近上限就停止新调度”的需求。

备选方案：
- 仅使用总额度比例：对仅配置日/周额度的账号无效。
- 分别配置多个阈值：增加理解和运营成本，超出当前需求。

### 决策 3：引入与窗口费用一致的“三态调度”语义

账号使用量阈值新增三种调度结果：
- `schedulable`：未达到阈值，可参与新调度。
- `sticky_only`：达到阈值，但当前仍有活跃会话，仅允许原会话继续。
- `not_schedulable`：达到阈值，且当前无活跃会话，不再参与调度。

调度快照只返回 `schedulable` 账号用于新分配；网关在命中已有粘性账号时，允许 `sticky_only` 账号继续服务当前会话。

原因：
- 现有窗口费用控制已经验证了“新会话停止、粘性会话可继续”的模式，复用同类语义最稳妥。
- 三态比单纯布尔值更容易覆盖“正在对话中”和“对话已结束”两个阶段。

备选方案：
- 达阈值后立即下线账号：会中断正在进行的对话，不符合需求。
- 继续允许新会话直到人工停用：无法实现接近上限时止损。

### 决策 4：活跃会话状态沿用现有 SessionLimitCache 作为“会话未结束”判据

是否仍允许阈值账号继续服务当前对话，不通过新状态表维护，而是沿用现有 `SessionLimitCache` 活跃会话计数。只要账号在会话缓存中的活跃数量大于 0，就视为仍有未结束会话；当计数归零后，该账号在下一次调度判断中立即转为 `not_schedulable`。

原因：
- 现有网关已经用这套缓存处理会话数量限制和粘性续用，复用成本最低。
- “对话结束”在现有系统里本就通过活跃会话超时自然衰减判定，保持一致能避免引入两套结束标准。

备选方案：
- 新建专门的会话生命周期表：更精确，但实现和维护成本高。
- 仅按请求完成瞬间切换状态：流式/多轮会话场景下容易误判。

### 决策 5：管理端同时暴露阈值配置与账号当前命中状态

分组管理接口新增阈值读写字段，列表/详情需要返回该值；账号可调度性展示处应补充“因使用量阈值仅允许粘性会话”或“因使用量阈值停止调度”的状态，以便运营排查为什么账号未被继续分配。

原因：
- 只有配置入口没有状态反馈，会导致阈值命中后的行为难以解释。
- 与现有 schedulable、window cost 等状态展示并列，便于统一运维心智。

备选方案：
- 只做后端规则不做可视化：短期实现简单，但排障成本高。

## 风险 / 权衡

- [按最大使用比例判定会比只看总额度更严格] → 在配置说明中明确“任一已启用配额维度达到阈值都会触发”。
- [活跃会话结束依赖缓存超时，不是强实时事件] → 复用现有 idle timeout 机制，并在说明中明确“结束”按现有会话活跃判定处理。
- [账号跨多个分组时行为可能不一致] → 明确规则为“按请求分组独立生效”，避免隐式全局封禁。
- [所有账号都达到阈值时用户仍会失败] → 沿用现有无可用账号错误，并建议运营通过监控提前调整阈值或补充账号。

## Migration Plan

1. 为分组模型、仓储、DTO 与管理接口增加 `account_usage_threshold_percent` 字段及校验。
2. 在账号调度判定中新增“使用量阈值”评估函数，并接入调度快照与网关粘性账号判断。
3. 在管理端分组配置和账号状态展示中增加阈值表单、文案和命中原因。
4. 补充后端测试，覆盖新调度、粘性续用、会话结束后摘除和无配额上限账号等场景。
5. 灰度上线后观察高使用量账号的切换行为，再决定是否补充更细的监控或告警。
