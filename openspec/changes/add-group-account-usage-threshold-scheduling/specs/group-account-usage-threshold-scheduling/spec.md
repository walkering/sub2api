## 新增需求

### 需求:管理员可以按分组配置账号使用量调度阈值
系统必须允许管理员为每个分组配置账号使用量调度阈值百分比，用于控制该分组内账号何时停止接收新的调度。阈值必须支持百分比表达，空值或小于等于 0 的值必须视为关闭该控制。

#### 场景:为分组设置 95% 阈值
- **当** 管理员为分组保存账号使用量调度阈值 `95`
- **那么** 系统必须把该分组的阈值保存为启用状态
- **并且** 后续该分组调度账号时必须以 95% 作为命中阈值

#### 场景:关闭分组阈值控制
- **当** 管理员清空阈值或提交小于等于 0 的值
- **那么** 系统必须关闭该分组的账号使用量调度控制

### 需求:系统必须按已配置配额维度计算账号使用量比例
系统必须基于账号已有配额字段计算使用量比例。对于 `quota_limit`、`quota_daily_limit`、`quota_weekly_limit` 中分母大于 0 的维度，系统必须分别计算“已用量/上限”，并以其中最大比例作为该账号在当前分组下的使用量比例；没有任何可用上限的账号必须视为不受该阈值控制。

#### 场景:总额度达到阈值
- **当** 账号 `quota_limit` 为 100 且 `quota_used` 为 95
- **并且** 所在分组阈值为 95%
- **那么** 系统必须判定该账号已达到使用量阈值

#### 场景:日额度先达到阈值
- **当** 账号 `quota_limit` 为 100 且 `quota_used` 为 60
- **并且** 账号 `quota_daily_limit` 为 20 且 `quota_daily_used` 为 19
- **并且** 所在分组阈值为 95%
- **那么** 系统必须按 95% 的日额度使用比例判定该账号已达到使用量阈值

#### 场景:账号没有可计算的配额上限
- **当** 账号未配置总额度、日额度和周额度上限
- **并且** 所在分组开启了账号使用量调度阈值
- **那么** 系统必须将该账号视为不受该阈值控制

### 需求:达到阈值的账号不得再参与新的调度
当账号在某个分组下的使用量比例大于等于该分组配置的阈值时，系统必须立即将该账号从该分组的新调度候选中移除，禁止再把新的用户对话分配到该账号。

#### 场景:新请求避开已达阈值账号
- **当** 分组内账号 A 已达到使用量阈值
- **并且** 分组内账号 B 仍可调度
- **当** 新的用户请求进入该分组且不存在账号 A 的活跃粘性会话
- **那么** 系统必须禁止把该请求分配给账号 A
- **并且** 必须优先从其他可调度账号中选择

### 需求:达到阈值的账号必须允许当前活跃对话继续
当账号达到阈值时，如果该账号仍承载当前用户的活跃对话，系统必须允许该对话继续使用原账号完成，禁止因达到阈值而主动中断这次对话。

#### 场景:对话进行中达到 95%
- **当** 用户当前对话已经绑定到账号 A
- **并且** 账号 A 在该对话进行期间达到所在分组的使用量阈值
- **并且** 账号 A 仍存在该对话对应的活跃会话
- **那么** 系统必须继续让该对话使用账号 A
- **并且** 禁止因为达到阈值而立即结束这次对话

### 需求:活跃对话结束后账号必须立即退出该分组调度
当达到阈值的账号不再有活跃会话时，系统必须立即将该账号标记为不可调度；同一用户后续发起的新一轮请求时，系统必须改选该分组内其他可用账号，而不是继续复用已达阈值的账号。

#### 场景:会话结束后切换到其他账号
- **当** 账号 A 已达到所在分组的使用量阈值
- **并且** 账号 A 的活跃会话数量已经归零
- **并且** 分组内仍有账号 B 可调度
- **当** 该用户随后再次向该分组发起新请求
- **那么** 系统必须禁止继续把请求调度到账号 A
- **并且** 必须把请求切换到账号 B 或其他仍可调度的账号
