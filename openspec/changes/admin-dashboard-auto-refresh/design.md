## 上下文

`frontend/src/views/admin/DashboardView.vue` 当前通过 `loadDashboardStats()` 与 `loadChartData()` 负责加载总览卡片、趋势图、用户趋势和消费排行，但触发方式只有首次挂载、日期范围变更、粒度切换和手动点击刷新。页面还没有自动刷新状态、倒计时展示和偏好持久化。

仓库里已经存在两类可复用模式：
- `frontend/src/views/admin/ops/OpsDashboard.vue`：提供基于 `useIntervalFn` 的倒计时和自动刷新节奏控制。
- `frontend/src/views/admin/AccountsView.vue`：提供前端本地持久化的自动刷新开关、预设间隔和下拉交互。

这次变更只覆盖 `/admin/dashboard`，不改动后端接口，也不引入新的系统设置。实现必须复用现有仪表盘查询接口，且不能因为自动刷新引入重叠请求或破坏当前日期范围 / 粒度筛选语义。

## 目标 / 非目标

**目标：**
- 在 `/admin/dashboard` 增加自动刷新开关、预设间隔选择和倒计时反馈。
- 自动刷新时复用现有数据加载逻辑，在当前 `startDate`、`endDate`、`granularity` 条件下刷新全部仪表盘数据。
- 将自动刷新偏好保存在浏览器本地，页面刷新后恢复上次配置。
- 避免自动刷新与手动刷新、筛选切换发生并发重叠请求。

**非目标：**
- 不新增后端接口、数据库字段或后台系统设置。
- 不把自动刷新做成全局后台配置；本次仅作用于 `/admin/dashboard`。
- 不重构现有图表组件或改写仪表盘查询协议。
- 不要求跨设备同步自动刷新配置。

## 决策

### 决策 1：自动刷新配置使用前端本地持久化，而不是后端设置

自动刷新启用状态与刷新间隔使用浏览器本地存储保存，并在 `DashboardView` 挂载时恢复。

原因：
- `/admin/dashboard` 当前没有对应的后端设置接口，新增后端配置会扩大变更范围。
- 自动刷新更偏向“当前管理员在当前浏览器里的使用偏好”，不要求跨设备共享。
- 仓库内 `AccountsView` 已有同类本地持久化模式，可直接复用思路。

备选方案：
- 使用后端系统设置：需要增加 API、DTO 和权限校验，超出本次需求。
- 使用 URL query 持久化：会污染分享链接，且不适合保存倒计时相关 UI 偏好。

### 决策 2：自动刷新统一复用现有全量加载入口

自动刷新触发时统一走 `loadDashboardStats()`，而不是拆分为只刷新部分区域的独立逻辑。

原因：
- 总览卡片、趋势图、用户趋势和排行本来就是同一页面语义上的同一批数据，自动刷新时保持一致更稳定。
- 直接复用现有加载入口可以减少分叉逻辑，降低手动刷新与自动刷新行为不一致的风险。
- 当前 `loadChartData()` 只覆盖图表相关数据，不适合作为自动刷新入口，否则顶部卡片会停留在旧值。

备选方案：
- 只刷新图表：请求量更低，但会让顶部卡片与图表更新时间不一致。
- 新增第三套“自动刷新专用加载函数”：可读性和维护成本都更差。

### 决策 3：倒计时与并发保护独立建模

在页面内部增加独立的自动刷新状态：启用开关、间隔秒数、倒计时和“刷新进行中”保护。倒计时每秒递减；当倒计时归零时，仅在当前没有刷新任务时才触发新的自动刷新。刷新成功后重置倒计时。

原因：
- `DashboardView` 现有的序列号只解决“旧响应覆盖新响应”，不能阻止多个刷新周期并发发起请求。
- 自动刷新与手动刷新、日期变更都可能命中同一批接口，必须有统一的 in-flight guard。
- `OpsDashboard` 已验证“倒计时驱动 + 成功后重置”的模式足够清晰。

备选方案：
- 仅依赖现有 `loading/chartsLoading/userTrendLoading`：状态分散，难以准确表达“整页刷新是否进行中”。
- 允许重叠请求再用序列号丢弃旧响应：会徒增请求量，且对后端无意义。

### 决策 4：自动刷新控件放在现有刷新入口附近，沿用管理后台现有交互习惯

自动刷新入口与现有“刷新”按钮放在同一区域，展示启用状态、倒计时和间隔选项；手动刷新入口保留。

原因：
- 当前日期范围卡片已经是页面的“数据控制区”，新增自动刷新放在这里最符合用户预期。
- `AccountsView` 的下拉式交互已被管理员页面使用，复用后学习成本低。
- 保留手动刷新能覆盖“临时关闭自动刷新但立即重载一次”的场景。

备选方案：
- 只用一个开关不提供间隔选择：实现最简单，但可控性不足。
- 把自动刷新塞进更多设置弹窗：多一步操作，不适合高频使用。

## 风险 / 权衡

- [自动刷新增加页面请求频率] → 默认关闭，并限制为预设间隔选项，避免无上限轮询。
- [页面加载较慢时重复触发刷新] → 使用统一的刷新进行中保护，禁止自动刷新并发重入。
- [本地存储配置损坏或缺失] → 读取失败时回退到“关闭 + 默认间隔”的安全状态。
- [手动刷新和自动刷新倒计时语义不一致] → 所有成功刷新入口统一重置倒计时。

## Migration Plan

1. 仅修改前端 `DashboardView` 与相关文案、测试文件。
2. 发布后无需数据迁移；旧页面访问行为保持不变，未开启自动刷新时仍是纯手动刷新。
3. 若上线后发现问题，可直接回退前端改动，不影响后端兼容性。

## Open Questions
