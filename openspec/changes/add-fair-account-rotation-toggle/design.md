## 上下文

当前系统已经具备完整的账号调度基础能力，包括账号优先级、平台与模型过滤、额度与窗口成本检查、会话限制以及粘性会话绑定。实际问题不在于“能否选到可用账号”，而在于新请求在大量账号并存时容易长期集中落到少数账号上，造成其他账号长期得不到使用。

本次变更要求在系统设置中引入一个显式开关，允许管理员在“保持当前行为”与“雨露均沾”之间切换。该变更会同时影响后端设置层、多个账号选择入口和前端系统设置页，属于跨模块行为调整，适合先形成设计文档以统一实现边界。

## 目标 / 非目标

**目标：**
- 新增 `account_fair_rotation_enabled` 对应的系统设置开关，并在系统设置中可读可写。
- 开关关闭时，所有账号选择入口必须保持当前行为，作为兼容模式。
- 开关开启时，仅对“已通过现有筛选规则且属于当前最高可选优先级”的候选账号集启用公平轮转。
- 公平轮转必须只影响新的选账号过程，不破坏已有粘性会话的账号复用语义。
- 新设计必须覆盖 `GatewayService`、`OpenAIGatewayService` 和 `GeminiMessagesCompatService` 等当前存在独立选择逻辑的入口。

**非目标：**
- 不改变账号 `priority` 的含义，也不让低优先级账号因为公平轮转而抢占高优先级账号。
- 不改变额度阈值、RPM、窗口成本、模型路由、平台限制或粘性会话 TTL 的现有规则。
- 不把公平轮转扩展为加权调度、动态负载均衡或后台可配置的复杂算法。
- 不解决所有历史调度倾斜问题；本次仅针对开启开关后的新增请求分配行为。

## 决策

### 决策 1：用独立系统设置开关控制公平轮转，默认关闭

新增一个布尔设置键，由后端设置服务负责默认值和持久化，前端系统设置页提供开关展示与保存。默认值为关闭，保证升级后不改变当前生产行为。

原因：
- 需求明确要求“关闭则原有规则，开启则雨露均沾规则”。
- 独立设置键便于回滚、灰度和审计，不会和其他调度特性耦合。

备选方案：
- 直接替换现有选择算法：风险高，无法给已有部署保留兼容模式。
- 使用环境变量或编译期开关：不便于运营动态切换，也无法通过现有系统设置接口管理。

### 决策 2：公平轮转只作用于当前最高可选优先级的候选账号集

调度流程先沿用现有所有过滤和排序前置条件，得到“当前请求可用的账号集合”。如果存在多个优先级，仍然先锁定当前最高优先级的账号子集；公平轮转只在这个子集内部生效，不跨优先级扩散。

原因：
- 这样可以保证现有 `priority` 语义完全不变，降低行为回归风险。
- 用户诉求是“雨露均沾使用账号”，而不是“打破既有优先级策略”。

备选方案：
- 在全部可用账号上直接轮转：会让低优先级账号越级被选中，破坏现有调度意图。
- 先做加权打散再统一排序：复杂度更高，收益不足。

### 决策 3：公平轮转对“新选择”使用组级轮转游标，而不是纯随机或继续依赖 `LastUsedAt`

开启公平轮转后，系统对当前最高优先级候选账号生成一个稳定顺序，并结合一个按分组维度维护的轮转游标选择下一个账号；成功选中后游标推进到下一个位置。若候选集变化导致当前游标落点失效，则跳过失效账号并继续向后寻找。只有在没有可用游标信息时，才从当前顺序的首个候选开始。

原因：
- 用户问题的核心是避免“一个账号先被吃满，再切下一个”。显式轮转游标比随机更能保证连续请求的覆盖面。
- 单纯依赖 `LastUsedAt` 在实际并发和粘性复用下容易继续向少数账号倾斜，不能稳定满足“雨露均沾”目标。

备选方案：
- 纯随机：分布可能改善，但无法保证短期内覆盖多个账号。
- 继续沿用 `LastUsedAt` 只是修改 tie-break：只能缓解，不能保证轮转公平性。

### 决策 4：公平轮转状态按“调度作用域”存储，至少隔离到分组

轮转游标必须与调度作用域绑定，最小粒度为分组；在存在模型路由、平台拆分或混合调度时，作用域键还应包含会影响候选集稳定性的必要上下文，避免不同路由结果共享同一个游标导致跳跃。

原因：
- 同一个分组下不同路由集合可能完全不同，共享游标会破坏公平性并增加调试难度。
- 作用域化后可以保证轮转仅在相同候选语境中推进。

备选方案：
- 全局单游标：会把不相关分组和路由混在一起，语义错误。
- 每次临时根据内存状态决定下一个账号：跨实例不稳定，难以解释行为。

### 决策 5：粘性会话保持最高优先级，公平轮转只用于未命中粘性会话的新请求

请求若先命中有效粘性会话，并且绑定账号仍满足现有各项可调度检查，则继续使用原账号；只有未命中粘性会话、粘性账号失效或需要重新选择账号时，才进入公平轮转逻辑。

原因：
- 粘性会话的价值在于对话连续性和上游上下文稳定性，不能被新调度模式破坏。
- 用户诉求针对“账号额度消耗分配”，并未要求强制打断已有会话。

备选方案：
- 开启公平轮转后忽略粘性会话：会直接改变现有对话行为，风险不可接受。
- 让粘性会话也参与轮转：语义自相矛盾。

### 决策 6：多个选择入口复用同一套“公平轮转候选选择”辅助逻辑

`GatewayService`、`OpenAIGatewayService` 和 `GeminiMessagesCompatService` 当前各自维护一部分基于优先级与 `LastUsedAt` 的选择流程。实现时应抽取共享的公平轮转决策辅助逻辑，在各入口完成各自过滤后统一调用，避免不同入口出现“一个入口轮转、另一个入口继续吃单号”的不一致现象。

原因：
- 用户感知的是系统整体账号使用是否均衡，不会接受不同入口行为不一致。
- 抽出共享逻辑更利于回归测试和后续维护。

备选方案：
- 每个入口各自实现轮转：容易产生细节漂移，后续难以修正。

## 风险 / 权衡

- [轮转游标需要跨请求保存状态] -> 复用现有缓存/存储基础设施，按作用域保存轻量游标信息，并在缺失时安全降级到当前顺序首项。
- [候选集在高并发下频繁变化，严格轮转可能被打断] -> 接受“尽量公平而非绝对平均”的语义，只要求避免长期只消耗单个账号。
- [多个入口当前选择代码不完全一致] -> 通过共享辅助函数和覆盖测试统一关键行为，减少入口差异。
- [开启后调度顺序会变化，可能影响既有流量分布] -> 用默认关闭和显式开关降低升级风险，并提供可快速回退的兼容模式。
- [新账号或 `LastUsedAt=nil` 账号可能集中插队] -> 公平轮转基于当前候选稳定顺序推进，新账号会参与轮转，但不会持续霸占所有后续请求。

## Migration Plan

1. 新增系统设置键并设默认值为关闭，完成后端接口和前端设置页接线。
2. 在各账号选择入口引入统一的公平轮转辅助逻辑，但仅在读取到开关开启时启用。
3. 部署后默认仍为旧行为；管理员保存开启后，新请求开始按公平轮转模式选择账号。
4. 如需回滚，只需关闭该设置开关；保留新增设置键和游标状态不会影响旧模式继续运行。
