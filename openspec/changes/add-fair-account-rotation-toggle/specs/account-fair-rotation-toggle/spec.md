## 新增需求

### 需求:管理员可以配置账号公平轮转开关
系统必须允许管理员在系统设置中开启或关闭账号公平轮转调度。该开关默认必须为关闭；关闭时系统必须保持现有账号调度行为不变。

#### 场景:读取默认关闭的公平轮转开关
- **当** 系统初始化默认设置且管理员尚未修改账号公平轮转配置
- **那么** 系统必须返回 `account_fair_rotation_enabled=false`

#### 场景:管理员开启公平轮转开关
- **当** 管理员在系统设置中将账号公平轮转开关设为开启并保存
- **那么** 系统必须保存该配置
- **并且** 后续新的账号调度必须按公平轮转规则执行

#### 场景:管理员关闭公平轮转开关
- **当** 管理员在系统设置中将账号公平轮转开关设为关闭并保存
- **那么** 系统必须保存该配置
- **并且** 后续新的账号调度必须继续按现有调度规则执行

### 需求:开启后系统必须在同优先级可调度账号之间公平轮转
当 `account_fair_rotation_enabled=true` 时，系统必须在所有既有可调度性检查完成后，仅对当前最高可选优先级内的候选账号执行公平轮转。连续进入的新的非粘性请求必须依次覆盖多个可用账号，禁止长期只重复消耗单个账号直到其额度耗尽后才切换。

#### 场景:多个同优先级账号轮转承接新的请求
- **当** 账号 A、B、C 处于同一分组
- **并且** 账号 A、B、C 拥有相同优先级
- **并且** 账号 A、B、C 都通过当前请求的现有可调度性检查
- **并且** 账号公平轮转开关已开启
- **并且** 连续进入多个彼此独立且未命中粘性会话的新请求
- **那么** 系统必须按公平轮转规则把这些请求分配到 A、B、C 等多个账号
- **并且** 禁止在 A、B、C 持续可用时长期只重复选择其中一个账号

#### 场景:只有一个可调度账号时继续使用该账号
- **当** 某个请求在完成现有筛选后只剩账号 A 一个可调度候选
- **并且** 账号公平轮转开关已开启
- **那么** 系统必须选择账号 A

### 需求:公平轮转不得破坏现有优先级与粘性会话语义
账号公平轮转必须被视为当前调度规则之上的附加分配策略。无论开关是否开启，系统都必须先遵守现有优先级、模型路由、平台限制、额度限制和粘性会话规则；公平轮转只允许在需要重新选择新账号时重新分配当前最高优先级的候选账号。

#### 场景:高优先级账号仍先于低优先级账号参与选择
- **当** 账号 A 与账号 B 都通过当前请求的现有可调度性检查
- **并且** 账号 A 的优先级高于账号 B
- **并且** 账号公平轮转开关已开启
- **那么** 系统必须先在账号 A 所在的更高优先级候选集中完成选择
- **并且** 禁止仅因为公平轮转而直接把该请求分配给账号 B

#### 场景:粘性会话继续复用原账号
- **当** 某个用户的当前会话已经绑定到账号 A
- **并且** 账号 A 仍满足该请求的现有粘性复用条件
- **并且** 账号公平轮转开关已开启
- **那么** 系统必须继续复用账号 A
- **并且** 禁止仅因为公平轮转而把该会话切换到账号 B 或其他账号

#### 场景:关闭开关时保持原有调度规则
- **当** 账号公平轮转开关为关闭
- **并且** 某个新请求进入账号调度流程
- **那么** 系统必须继续使用当前已有的账号选择规则
- **并且** 禁止因为本次变更引入新的公平轮转行为
