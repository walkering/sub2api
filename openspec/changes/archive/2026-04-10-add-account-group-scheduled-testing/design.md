## 上下文

账号管理已具备单账号定时测试与即时测试能力，但仅限逐个配置或手动触发。分组管理缺少定时测试与立即测试入口，也没有统一的测试执行进度视图。随着账号池规模增长，管理员需要面向分组进行周期性巡检，同时通过错峰策略避免大量账号在同一时刻触发测试。

后端已有定时测试调度器与账号测试执行服务，但当前并未覆盖“按分组批量计划 + 分批错峰执行 + 实时进度”的完整链路。

## 目标 / 非目标

**目标：**
- 在账号管理与分组管理提供分组定时测试入口。
- 定时测试计划支持批次大小与偏移量，触发时按批次错峰执行。
- 在分组管理提供“测试账号（立即）”入口，创建后台任务并限制并发为 5。
- 提供实时日志或进度流，便于管理员查看执行情况。

**非目标：**
- 不重写现有账号测试逻辑或测试结果判定。
- 不引入跨分组/跨类型的混合测试策略。
- 不提供复杂的任务调度编排（如依赖、优先级链）。
- 不改变现有单账号定时测试的用户路径。

## 决策

### 决策 1：新增“分组定时测试计划”模型而非为每个账号生成计划

分组定时测试以分组计划为单位存储，计划包含 `group_id`、时间规则（沿用现有定时测试格式）、`batch_size`、`offset` 等字段。触发时由服务端解析计划并按分组筛选账号。

原因：
- 运营上关注的是分组的周期性巡检，而不是每个账号独立计划。
- 计划数量随账号数线性增长会导致维护与存储成本过高。

备选方案：
- 为每个账号生成独立计划：可行但管理成本高。
- 客户端定时触发：不可靠且难以统一调度。

### 决策 2：分组定时测试按稳定顺序分批，并用偏移量错峰

计划触发时，系统按稳定顺序（例如账号 ID 或创建时间）确定账号列表，再按 `batch_size` 分批。各批次启动时间以 `offset` 间隔递增，避免同一时间集中测试。

原因：
- 稳定顺序便于排查与复现。
- 批次间偏移能控制负载峰值。

备选方案：
- 随机排序：结果不可预测。
- 按每个账号设置偏移：配置成本过高。

### 决策 3：立即测试以“后台任务 + 5 并发工作池”执行

分组“测试账号（立即）”会创建后台任务，任务内维护账号队列与工作池（并发上限 5）。当某个账号测试完成后，自动从队列补位执行下一账号，直到队列为空。

原因：
- 避免 HTTP 同步执行导致超时。
- 固定并发上限易于控制资源占用。

备选方案：
- 同步批量调用：易超时，且难以限制并发。
- 为每个账号创建独立任务：日志与进度聚合困难。

### 决策 4：实时日志以 SSE 方式提供

实时日志采用 SSE（Server-Sent Events）按任务 ID 推送进度与结果；客户端可在“实时日志”入口查看流式更新，同时提供最新进度的快照接口用于断线恢复。

原因：
- SSE 实现简单、服务端负担较小，适合单向进度流。
- 断线重连容易恢复，并能与现有 HTTP 鉴权复用。

备选方案：
- WebSocket：更重的连接管理。
- 轮询：实时性与负载不佳。

## 风险 / 权衡

- [批次错峰可能延长整体测试完成时间] → 在界面明确展示预计耗时，并允许配置较小的 offset。
- [并发上限 5 对大分组执行较慢] → 后续可通过配置提升上限或拆分分组执行。
- [实时日志存储增长] → 限制日志保留时长与最大条数，超限后保留摘要。

## Migration Plan

1. 后端增加分组定时测试计划存储与 API。
2. 后端增加测试任务与实时日志 API，并在执行器接入。
3. 前端在账号管理与分组管理新增入口与表单。
4. 灰度启用后验证执行负载与日志稳定性，必要时调整默认 batch/offset。

- 实时日志需要保留多长时间，是否支持下载或导出？
