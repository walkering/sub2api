## 上下文

当前账号页已经支持三类相关能力：
- 通过 `AccountBulkActionsBar` 对已勾选账号执行重置状态、刷新令牌和批量编辑。
- 通过 `BulkEditAccountModal` 使用勾选开关决定哪些字段参与更新，未勾选字段不会落库。
- 后端已经具备 `ListByGroup`、`BindGroups`、`BulkUpdate` 等账号分组与批量更新基础能力，但它们只支持“对给定账号集合统一处理”，不支持“按分组范围缩小执行对象”或“从源分组按数量迁移账号”。

这次需求同时涉及前端批量操作交互、后端批量接口语义和账号运行态校验。实现时必须避免把“操作作用范围”与“批量编辑里要修改哪些字段”混在一起，且不能破坏现有批量编辑勾选语义。

## 目标 / 非目标

**目标：**
- 新增一个独立的账号分组迁移流程，允许管理员指定源分组、目标分组和迁移数量。
- 在服务端校验迁移资格：仅允许同类型、状态正常、未限流、未被占用的账号被迁移。
- 为重置状态、刷新令牌和批量编辑增加按分组范围执行的能力。
- 保持现有 `BulkEditAccountModal` 的字段勾选语义不变，未勾选字段继续保持原值。

**非目标：**
- 不改变账号分组的数据库结构，也不引入新的分组实体。
- 不重做现有账号列表筛选、排序和选择模型。
- 不扩展到删除、启停调度等其他批量操作；本次仅覆盖重置状态、刷新令牌和批量编辑。
- 不自动补足不足数量的可迁移账号；当可迁移数量不足时直接失败。

## 决策

### 决策 1：分组迁移使用独立接口，而不是复用现有 bulk update

新增独立的“账号分组迁移”请求模型与接口，输入至少包含 `source_group_id`、`target_group_id` 和 `count`。它不复用现有 `/admin/accounts/bulk-update`，因为迁移流程不是“给定账号 ID 后统一改字段”，而是“服务端先从源分组选择符合资格的账号，再执行移动”。

原因：
- 迁移动作需要服务端自己筛选候选账号、校验资格并按稳定顺序截取前 N 个账号。
- 现有 `bulk-update` 中的 `group_ids` 表示“把指定账号绑定成哪些分组”，语义与“从源组挪出多少个账号”不同，强行复用会混淆请求含义。

备选方案：
- 在 `bulk-update` 中新增 `source_group_id` / `target_group_id`：会让“字段更新”和“服务端选账号再迁移”耦合在一个接口里，难以校验与测试。
- 前端先查出候选账号再逐个调用单账号编辑：并发差、不可原子校验，也无法保证数量选择的一致性。

### 决策 2：按分组执行的批量操作统一引入 `scope_group_id`

对批量重置状态、批量刷新令牌和批量编辑统一新增可选的 `scope_group_id` 语义。当前端传入该字段时，后端必须先用“当前勾选账号 ID ∩ 属于该分组的账号”得到实际执行集合，再执行原有批量操作。

原因：
- `scope_group_id` 只描述“作用范围”，不会和 `bulk-update` 中已有的 `group_ids`（目标绑定分组）冲突。
- 这样可以同时满足“我勾选了一批账号，但只想对某个分组里的那部分执行操作”的需求。

备选方案：
- 直接复用 `group_ids` 作为范围字段：会与批量编辑里的“要把账号绑定到哪些分组”发生歧义。
- 在前端过滤完账号后只传更小的 `account_ids`：前端可见状态不可信，服务端仍需要再次校验分组归属，重复实现更多。

### 决策 3：迁移资格由服务端统一判定，并按稳定顺序截取

迁移候选账号由服务端根据源分组查询结果统一判定，至少包含以下条件：
- 账号类型必须一致；若候选集中存在多个 `type`，系统必须拒绝本次迁移。
- 账号必须处于可正常使用状态（如 `status=active` 且未处于需排除的异常/暂停状态）。
- 账号不得处于限流窗口中。
- 账号不得处于运行占用中；运行占用基于现有实时运行态来源判断，如并发占用、活跃会话等。

当符合条件的账号数量不少于 `count` 时，系统按现有分组内稳定顺序筛选前 N 个执行迁移；若不足，则整次请求失败，不做部分迁移。

原因：
- 资格校验必须以服务端实时状态为准，不能信任前端展示数据。
- “不足则失败”比“部分迁移”更容易让管理员理解，也更容易回放和测试。
- 复用现有分组内顺序可以避免每次随机选账号带来的不可预测性。

备选方案：
- 前端先筛出可迁移账号再提交：无法保证与服务端实时状态一致。
- 可迁移多少就迁移多少：会让“移动 10 个账号”变成不确定结果，影响运营预期。

### 决策 4：分组迁移接口对非法输入与缺失分组快速失败

在进入候选账号筛选前，服务端必须先校验 `source_group_id > 0`、`target_group_id > 0`、`count > 0`、`source_group_id != target_group_id`，并验证源分组和目标分组都真实存在。任一条件不满足时，请求必须直接失败，不进入可迁移账号计算。

原因：
- 这些约束属于请求本身是否合法，而不是业务筛选结果，应该尽早返回明确错误。
- 先做 fail-fast 校验能避免把“参数错误”混淆成“可迁移账号不足”，也能减少无意义的分组查询和运行态判断。

备选方案：
- 允许源组与目标组相同并返回 0 条迁移：会掩盖误操作。
- 允许 `count<=0` 并由后端静默纠正：不利于前后端统一校验和测试。
- 把分组不存在也视为“数量不足”：错误原因不准确，影响排障。

### 决策 5：批量编辑的字段勾选逻辑保持原样，分组范围只是额外过滤

`BulkEditAccountModal` 现有的 enable 开关继续决定哪些字段进入请求；新增的分组范围选择只影响“哪些账号会被更新”，不改变“哪些字段会被修改”。

原因：
- 用户明确要求“原有的勾选进行设置也不影响保持原样”。
- 这样能把“账号范围”和“字段变更集合”解耦，降低误操作风险。

备选方案：
- 在分组模式下重写一套批量编辑表单：维护成本高，也容易与现有弹窗行为分叉。

## 风险 / 权衡

- [“正常账号”的定义与现有运行状态来源不完全重合] -> 在实现中把迁移资格收敛成单一服务端判定函数，并补充单元测试覆盖 active / rate-limited / in-use / temp-unschedulable 等边界。
- [批量编辑已有 `group_ids` 字段，新增分组范围后容易产生概念混淆] -> 请求模型明确使用单独的 `scope_group_id`，前端文案也区分“作用范围分组”和“要设置的账号分组”。
- [按数量迁移若中途部分失败会导致结果不一致] -> 在服务端先完整计算候选集并校验数量，再统一执行迁移；失败时返回明确的可迁移数量与失败原因。
- [批量操作入口变多后界面复杂度上升] -> 仅在相关操作前展示轻量级分组选择，不改变账号列表主流程。

## Migration Plan

1. 为账号管理接口增加新的分组迁移入口，并为现有批量操作请求增加可选的 `scope_group_id` 参数。
2. 前端在账号列表批量操作栏和批量编辑弹窗中增加分组范围选择与迁移入口。
3. 部署后无需数据迁移；未使用新参数的旧调用继续保持现有行为。
4. 如需回滚，只需回退前后端代码；数据库中的账号分组绑定关系仍沿用现有表结构。

## Open Questions

- 暂无阻塞性开放问题；实现时只需进一步确认“正在使用”在各平台上的统一判定来源（并发占用、活跃会话或其他运行态缓存）。
