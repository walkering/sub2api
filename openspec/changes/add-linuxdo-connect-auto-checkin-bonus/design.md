## 上下文

系统现有 LinuxDo Connect 能力已经覆盖登录和邀请码补全注册，但没有签到能力，也没有渠道专属的登录奖励机制。LinuxDo Connect 首次注册赠送订阅由独立变更维护；本设计要求自动签到奖励在该赠送订阅能力已归档、尚未归档或未开启时都能独立工作。用户这次希望在 LinuxDo Connect 登录成功后自动签到、弹出成功提示，并赠送 `1-5` 金额，同时要求具备后台开关。

LinuxDo Connect 登录成功目前存在两条主要路径：
- OAuth 回调直接完成登录并重定向前端
- 邀请码模式下先返回 `pending_oauth_token`，再由前端调用 `complete-registration` 完成建号和登录

设计必须同时覆盖这两条路径，并确保前端只在“本次登录实际触发签到奖励”时提示成功。

## 目标 / 非目标

**目标：**
- 新增 `linuxdo_connect_auto_checkin_bonus_enabled` 设置，默认值为 `false`。
- 开关开启后，LinuxDo Connect 登录成功自动执行当日签到检查。
- 单用户同一自然日内最多成功一次自动签到奖励。
- 首次成功签到时，服务端随机赠送 `1-5` 余额，并将结果回传给前端。
- 前端仅在本次自动签到成功时展示签到成功与实际奖励金额。

**非目标：**
- 不为普通邮箱登录、管理员登录或其他 OAuth 登录入口增加自动签到奖励。
- 不把奖励金额范围做成后台可配置项；本次固定为服务端随机 `1-5`。
- 不补做一个通用“全站签到中心”或签到历史页面。
- 不改变现有 LinuxDo Connect 赠送订阅逻辑。

## 决策

### 决策 1：新增 LinuxDo 专属布尔开关设置，默认关闭

新增 `SettingKeyLinuxDoConnectAutoCheckinBonusEnabled`，在设置模型、DTO、后台设置接口与管理页中暴露 `linuxdo_connect_auto_checkin_bonus_enabled` 字段，默认值为 `false`。

原因：
- 需求明确要求可开关控制，且默认不应改变现有登录行为。
- 独立设置键语义清晰，不会和 LinuxDo Connect 启用状态或赠送订阅设置混淆。

备选方案：
- 复用 LinuxDo Connect 总开关：无法区分“允许登录”和“允许登录后送钱”两件事。
- 通过环境变量硬编码控制：不利于后台运营切换，也无法被现有设置接口读取。

### 决策 2：自动签到采用“每日一次”的业务语义，而不是“每次登录都奖励”

开启开关后，LinuxDo Connect 登录成功只会触发一次“当日签到尝试”。若用户在当前服务端自然日已经完成过 LinuxDo 自动签到，则本次登录继续成功，但不重复奖励，也不返回“签到成功”状态。

原因：
- 这满足“登录成功自动签到”的产品诉求，同时规避反复登录刷奖励的明显风险。
- “签到”天然更适合按日限次，而不是按登录次数累计。

备选方案：
- 每次登录都奖励：风险过高，且会造成不可控的余额损耗。
- 首次注册时才奖励：覆盖范围过窄，不符合“登录成功自动签到”的字面需求。

### 决策 3：使用独立签到奖励记录做幂等与审计，而不是只在用户表增加一个日期字段

实现上新增 LinuxDo 自动签到奖励记录，至少保存 `user_id`、`reward_date`、`bonus_amount`、`source` 与时间戳，并对 `(user_id, reward_date, source)` 建立唯一约束。服务在事务中先写入签到记录，再增加用户余额；若唯一约束冲突，则视为今日已签到。

原因：
- 需要既解决“每日一次”的并发幂等问题，也保留奖励金额审计能力。
- 记录实际奖励金额后，后续排查投诉或对账时有依据。

备选方案：
- 在用户表增加 `last_linuxdo_auto_checkin_date`：实现更轻，但无法保存奖励金额，审计能力弱，并且未来若扩展其他签到来源会继续膨胀用户表。
- 只依赖缓存做日级限次：跨实例和重启后不可靠，不适合资金类奖励。

### 决策 4：奖励金额由服务端随机生成 `1-5` 整数，并与余额更新放在同一事务内

自动签到成功时，由服务端生成闭区间 `1-5` 的随机整数金额，并在同一事务中完成奖励记录写入与用户余额增加。事务成功后再刷新余额缓存或触发相关缓存失效。

原因：
- 奖励金额必须以服务端为准，避免前端伪造。
- 记录写入和余额增加在同一事务内，能保证“有记录就有余额变化”的一致性。

备选方案：
- 前端生成随机数后回传：不可信。
- 先加余额再写记录：并发失败时可能产生实际发钱但没有签到记录的问题。

### 决策 5：OAuth 成功结果增加自动签到结果字段，前后端按“是否实际奖励”决定提示

在认证服务层增加结构化返回，例如 `AutoCheckinResult { Awarded bool, BonusAmount int }`。LinuxDo OAuth 直接回调路径将其编码到前端重定向参数中；`complete-registration` 路径则在 JSON 响应中返回同样语义的字段。前端回调页在 `Awarded=true` 时弹出“签到成功，赠送 X 金额”，否则保持现有登录成功提示。

原因：
- 前端需要明确知道“本次是否真的签到成功”，仅靠登录成功无法区分。
- 两条成功路径都返回相同语义，便于前端复用判断逻辑。

备选方案：
- 始终弹“登录成功”：无法体现签到奖励是否发放。
- 在前端再次调用单独查询接口：增加一次请求，而且会把一次性登录结果变成不稳定状态。

### 决策 6：自动签到奖励异常采用 fail-close，已签到冲突视为未奖励

当自动签到奖励开关已开启且本次登录应执行签到时，服务端必须把“奖励记录写入 + 余额增加”视为登录成功链路中的必需步骤。除“同一用户当天已签到”这一幂等冲突外，若奖励记录创建失败、余额更新失败、事务提交失败或相关依赖不可用，则整个 LinuxDo 登录流程必须失败，不返回成功登录结果。

原因：
- 奖励属于登录成功后立即生效的资金类副作用，若登录已成功但奖励状态未知，会造成前端提示、余额和审计记录三者不一致。
- 现有返回契约需要精确表达“本次是否已奖励”；遇到异常时直接让登录失败，比返回一个语义不完整的“登录成功但奖励未知”更可预测。

备选方案：
- 登录成功但静默跳过奖励：用户会误以为系统吞掉奖励，且难以对账。
- 登录成功并返回“未奖励”：会把真正的系统异常伪装成正常业务分支，削弱监控与排障信号。

### 决策 7：自然日边界以服务端本地日期为准

“同一天”的判定以服务端运行时的本地日期为准，不额外引入新的时区配置。实现中统一在后端生成签到日期，避免前端时区参与判定。

原因：
- 当前系统没有独立的签到时区设置，引入新配置超出本次范围。
- 使用单一服务端日期来源能降低跨端时间差导致的重复领奖问题。

备选方案：
- 按浏览器本地时区判断：不可控且容易被篡改。
- 新增全局签到时区配置：扩展性更高，但超出本次提案目标。

## 风险 / 权衡

- [同一 LinuxDo 用户可能通过不同设备重复点击登录] -> 通过数据库唯一约束保证每日只能成功一次。
- [服务端本地日期与用户感知日期不一致] -> 在本次实现中接受该权衡，文案不承诺特定时区。
- [奖励发放新增持久化与事务逻辑] -> 成本高于简单加字段，但能换来幂等与审计能力。
- [LinuxDo 登录成功路径有两条，容易只改一条] -> 服务层统一返回自动签到结果，并为直接回调和邀请码补全都补测试。
- [LinuxDo 首次注册赠送订阅与自动签到奖励同时存在时容易被误认为强耦合] -> 明确两者是独立能力；自动签到奖励不以赠送订阅开关或归档状态为前提。

## Migration Plan

1. 新增 `linuxdo_connect_auto_checkin_bonus_enabled` 设置键，默认值设为 `false`。
2. 新增 LinuxDo 自动签到奖励记录的存储结构与唯一约束，用于保存每日奖励结果。
3. 部署后若管理员不启用该开关，系统保持现状，不自动签到、不增加余额、不弹奖励成功提示。
4. 若需回滚，保留新增设置键和值不会影响旧版本继续运行；新增签到记录表仅作为附加数据被忽略。
