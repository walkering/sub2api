## 新增需求

### 需求:管理后台总览指标必须支持按指标钻取
系统必须允许管理员从 `/admin/dashboard` 的关键统计指标和副指标直接进入对应的后台页面，并且必须让每个指标跳转到与其业务语义一致的目标页，而不是统一跳到单一路径。

#### 场景:点击正在使用的用户进入用户列表
- **当** 管理员在 `/admin/dashboard` 点击“正在使用的用户”指标
- **那么** 系统必须导航到 `/admin/users`
- **并且** 系统必须带上能够表示“今日活跃用户/正在使用的用户”的过滤参数
- **并且** 目标页首次加载时必须直接显示该过滤结果

#### 场景:点击账号异常数进入账号列表
- **当** 管理员在 `/admin/dashboard` 点击账号统计中的异常数量
- **那么** 系统必须导航到 `/admin/accounts`
- **并且** 系统必须让目标页以 `error` 状态过滤结果完成首次加载

#### 场景:点击今日请求进入用量列表
- **当** 管理员在 `/admin/dashboard` 点击“今日请求”指标
- **那么** 系统必须导航到 `/admin/usage`
- **并且** 系统必须带上当日的开始和结束日期过滤参数
- **并且** 目标页首次加载时必须显示与该日请求范围一致的请求记录

### 需求:用户管理列表必须支持仪表盘活动用户筛选
系统必须允许 `/admin/users` 读取来自仪表盘的活动范围和创建范围筛选参数，并且必须在首屏请求前把这些参数转换为后端过滤条件。

#### 场景:通过活动范围显示正在使用的用户
- **当** 管理员通过 `/admin/dashboard` 跳转到 `/admin/users?activity_scope=today_active`
- **那么** 系统必须在首次请求用户列表时应用“今日活跃用户”过滤
- **并且** 列表结果、总数和分页必须与该过滤条件保持一致

#### 场景:通过创建范围显示今日新增用户
- **当** 管理员通过 `/admin/dashboard` 跳转到 `/admin/users?created_scope=today`
- **那么** 系统必须在首次请求用户列表时应用“今日新增用户”过滤
- **并且** 管理员无需再次手工设置过滤条件

### 需求:系统必须提供管理员 API 秘钥列表作为钻取目标
系统必须提供 `/admin/api-keys` 管理页面及其后台列表接口，使仪表盘中的 API 秘钥统计可以跳转到管理员可操作的全局 API 秘钥列表。

#### 场景:点击全部 API 秘钥进入管理列表
- **当** 管理员在 `/admin/dashboard` 点击 API 秘钥总数
- **那么** 系统必须导航到 `/admin/api-keys`
- **并且** 目标页必须展示全局 API 秘钥列表

#### 场景:点击活跃 API 秘钥进入筛选结果
- **当** 管理员在 `/admin/dashboard` 点击活跃 API 秘钥数量
- **那么** 系统必须导航到 `/admin/api-keys?status=active`
- **并且** 目标页首次加载时必须只显示活跃 API 秘钥

### 需求:账号页和用量页必须恢复仪表盘传入的筛选条件
系统必须让 `/admin/accounts` 和 `/admin/usage` 在首次加载时解析来自仪表盘的 route query，并以这些 query 作为初始过滤条件。

#### 场景:账号页恢复状态筛选
- **当** 管理员通过仪表盘进入 `/admin/accounts?status=active`
- **那么** 系统必须在首次加载账号列表时应用 `active` 状态过滤
- **并且** 页面中的筛选控件必须与该状态保持一致

#### 场景:用量页恢复日期筛选
- **当** 管理员通过仪表盘进入带有 `start_date` 与 `end_date` 参数的 `/admin/usage`
- **那么** 系统必须在首次加载请求记录时应用这两个日期参数
- **并且** 页面中的日期筛选控件必须与 URL 中的日期保持一致
