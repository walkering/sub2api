## 上下文

`frontend/src/views/admin/DashboardView.vue` 当前已经展示 API 秘钥、账号、今日请求、用户等统计数据，也已经有“用户消费排行 -> 用量页”的单点跳转能力，但顶部统计卡片与副指标本身不可点击。管理员看到某个数字异常时，仍要自己切到 `/admin/users`、`/admin/accounts` 或 `/admin/usage` 手工输入筛选条件。

现状还有三个缺口：
- `/admin/users` 没有“正在使用的用户”筛选语义，也没有基于 route query 的首屏过滤恢复。
- `/admin/accounts`、`/admin/usage` 对仪表盘跳转所需的 query 恢复不完整。
- 系统没有全局管理员 API 秘钥列表页；当前只有按用户或分组查看 API 秘钥，以及单个 API 秘钥更新接口。

这次改动同时覆盖 dashboard、多个后台目标页、前端路由和部分后端管理接口，属于跨模块变更。

## 目标 / 非目标

**目标：**
- 让 `/admin/dashboard` 的关键统计指标都可以点击，并跳到对应后台页。
- 让目标页在首次加载时恢复仪表盘传入的筛选条件，不需要管理员再次手工输入。
- 为“正在使用的用户”提供与仪表盘口径一致的后台筛选能力。
- 补齐管理员 API 秘钥列表页与后台列表接口，承接仪表盘 API 秘钥钻取。

**非目标：**
- 不重做 dashboard 的整体视觉布局，只增强指标可点击性与跳转语义。
- 不把 dashboard 顶部所有数据都改成实时流式刷新；本次只处理导航和筛选钻取。
- 不重写现有用户、账号、用量页的全部筛选体系，只补足仪表盘跳转所需的 query 契约。
- 不改变 dashboard 现有统计数值的计算口径。

## 决策

### 决策 1：按“指标项”而不是“整张卡片”提供点击入口

总览页存在“一张卡片包含多个数字”的结构，例如账号卡同时包含总数、正常数、异常数。实现上应让每个可钻取的指标项独立可点击，而不是把整张卡片绑定到单一路由。

建议映射如下：
- `active_users` -> `/admin/users?activity_scope=today_active`
- `today_new_users` -> `/admin/users?created_scope=today`
- `total_users` -> `/admin/users`
- `total_api_keys` -> `/admin/api-keys`
- `active_api_keys` -> `/admin/api-keys?status=active`
- `total_accounts` -> `/admin/accounts`
- `normal_accounts` -> `/admin/accounts?status=active`
- `error_accounts` -> `/admin/accounts?status=error`
- `today_requests` -> `/admin/usage?start_date=<today>&end_date=<today>`

原因：
- 同一卡片里的不同数字代表不同业务语义，整卡跳转会丢失精度。
- 管理员希望点到哪个数字就看到哪个数字对应的列表，不需要二次筛选。

备选方案：
- 仅整卡可点击：实现简单，但无法同时承载总数/活跃数/异常数这类多目标跳转。
- 增加“查看详情”按钮：可读性更差，也会增加操作层级。

### 决策 2：目标页以 route query 作为钻取契约

dashboard 不通过全局 store 临时传参，而是通过明确的 URL query 把筛选条件传给目标页。目标页首次挂载时读取 query，并在首次请求前完成过滤状态初始化。

原因：
- URL 可分享、可刷新、可回退，适合作为跨页面导航契约。
- `UsageView` 已有部分 query 恢复能力，沿用这一模式更一致。
- 避免引入只为一次跳转存在的跨页全局状态。

备选方案：
- 使用 Pinia/内存态传递：刷新丢失，且不利于排查。
- 点击后仅跳到目标页不带筛选：不能满足“看见数字就进入对应结果集”的核心需求。

### 决策 3：正在使用的用户筛选必须走后端过滤

`/admin/users` 的“正在使用的用户”筛选需要在后端完成，而不是前端取回分页结果后再二次过滤。建议在用户列表过滤条件中新增面向仪表盘的活动范围字段，例如 `activity_scope=today_active`，并基于与 dashboard 相同的“今日有请求”口径进行查询。

原因：
- 用户列表本身是分页接口，前端后过滤会导致总数、分页和实际结果不一致。
- 仪表盘 `active_users` 已有稳定语义，列表页应复用同一业务口径。

备选方案：
- 前端根据用户详情或临时附加统计二次过滤：分页失真，不可接受。
- 仅跳转到 usage 页而不进入 users 页：不符合用户明确提出的“进入用户管理列表并过滤显示正在使用的用户”。

### 决策 4：API 秘钥钻取需要补齐新的管理员列表页，而不是复用现有局部入口

现有后台没有 `/admin/api-keys` 全局列表，也没有对应列表查询接口，因此仪表盘 API 秘钥指标必须新增一个管理员列表页及其后端列表/搜索接口。首版只要求承接总量和活跃量钻取，不要求一次性覆盖所有高级操作。

原因：
- 仪表盘是全局统计，目标页也必须是全局视角，按用户/分组局部查看无法承接。
- 后台已存在 API 秘钥更新接口与 usage 搜索辅助能力，说明此领域在管理侧是合理扩展点。

备选方案：
- 跳到 `/admin/usage` 通过 `api_key_id` 搜索：只能看请求记录，看不到 API 秘钥管理信息。
- 跳到用户详情里的 API 秘钥弹窗：无法承接“全部 API 秘钥”与“活跃 API 秘钥”。

### 决策 5：保留 dashboard 现有“今日”统计语义，不绑定到图表日期范围

`active_users`、`today_requests` 等卡片统计来自 dashboard 的今日快照，不随图表日期范围切换而改变。钻取时必须沿用这个“今日”语义，而不是错误地套用当前图表的 `startDate/endDate`。

原因：
- 这样目标页数据才会与卡片数字对齐。
- 仪表盘现有图表排行跳转仍可继续保留日期范围语义，两类交互不冲突。

备选方案：
- 统一按当前图表日期范围钻取：会让“今日请求”“正在使用的用户”失去与卡片值的一致性。

## 风险 / 权衡

- [用户活跃筛选涉及跨表或聚合查询，列表性能可能下降] -> 优先复用现有用量聚合口径；实现阶段若慢查询明显，再补索引或聚合查询优化。
- [新增管理员 API 秘钥列表页会扩大前后端改动面] -> 首版仅覆盖 dashboard 钻取所需的列表、状态筛选和基础搜索，不顺带做完整重构。
- [不同页面对 query 的字段命名不一致，容易造成导航契约碎片化] -> 在实现前先统一 query 命名并在页面首屏初始化阶段集中解析。
- [密集卡片内增加多个点击热点可能影响可读性] -> 通过 hover/下划线/图标提示只强化可钻取指标，不增加额外说明文本噪音。

## Migration Plan

1. 先补后端用户活动筛选和管理员 API 秘钥列表接口，再接入前端路由与页面。
2. 再为 dashboard 增加点击交互与 query 映射，并为 Users/Accounts/Usage/API Keys 页面补齐 query 恢复。
3. 这是一组向后兼容的后台增强；即使只发布后端或只发布前端，旧路径仍可继续使用，但完整体验需要前后端同时上线。
4. 如果上线后发现钻取目标或筛选语义不正确，可先回退 dashboard 点击入口；后端新增列表接口可保留，不影响兼容性。

## Open Questions

- 管理员 API 秘钥列表页首版需要暴露哪些列和操作，才能既满足钻取需求又不把范围扩大成完整重做。
- “正在使用的用户”在文案层是否直接统一显示为“今日活跃用户”，以减少与实时在线状态的歧义。
