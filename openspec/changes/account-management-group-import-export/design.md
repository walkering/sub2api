## 上下文

当前账号管理的导入与导出为全量操作，缺少按分组导出与导入指定分组的能力。在多分组运维场景下，管理员需要在导出后再手工筛选、或在导入后再手工分配，流程冗长且容易出错。新增分组维度可以在保持现有流程的同时覆盖更精细的管理需求。

## 目标 / 非目标

**目标：**
- 导出支持选择一个或多个分组，并仅导出所选分组中的账号。
- 导入支持选择目标分组，导入后的账号自动归入该分组。
- 明确分组选择与导入文件字段之间的优先级与校验规则。

**非目标：**
- 不改变导入导出文件的格式与字段结构（除非现有流程已有分组字段）。
- 不引入新的权限体系或角色模型，仅沿用现有权限校验。
- 不做跨分组的批量迁移能力（此变更只涉及导入/导出）。

## 决策

- 导出接口新增分组选项（多选）：后端按分组过滤数据，避免前端二次筛选。
- 导入接口新增目标分组参数（单选）：当指定目标分组时，导入的账号统一归入该分组。
- 若导入文件包含分组字段且用户选择了目标分组，以目标分组为最终归属，避免冲突。
- 默认行为保持向后兼容：未选择分组时保持现有导出行为；未指定目标分组时遵循现有导入流程（若系统已有默认分组规则则沿用）。

## 风险 / 权衡

- [分组误选导致数据归属错误] → 导入/导出界面增加明确提示与二次确认。
- [导出数据量过大影响性能] → 仍允许“全量导出”，但提示推荐使用分组导出。
- [参数兼容性] → 接口新增参数保持可选，确保老客户端仍可使用。

## 迁移计划

- 后端先支持新参数与校验逻辑，保持旧参数兼容。
- 前端发布分组选择 UI 后再逐步引导用户使用分组导出与导入。
- 出现问题时，可回退前端入口而保留后端兼容逻辑。

## 未决问题

- 导入导出是否需要在文件中增加“分组名称/ID”列以便审计？
- 导入失败的账号是否需要按分组生成更细粒度的错误报告？
